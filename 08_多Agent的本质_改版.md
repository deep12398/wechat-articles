# 多 Agent 的本质：把大问题拆给一群小工人

> 《工业级 AI Agent 开发实战》系列 · 第 08 篇

单 Agent 有天花板。

一个 Agent 要同时懂需求、查资料、写代码、跑测试、做审查、写报告，听起来很强，实际很脆。

上下文窗口会被撑爆，Prompt 角色会互相污染，工具太多会选错，任务越长越容易跑偏。

于是多 Agent 出现了。

但多 Agent 不是"一群 AI 讨论后涌现智能"。这个说法适合 demo，不适合生产。

工程上，多 Agent 就是多个单 Agent 加一套协作协议。

## 一、多 Agent 到底是什么

每个 Agent 本身都是完整的：有自己的 Prompt、Memory、Tools、循环和状态。

多 Agent 系统做的事，是把一个大任务拆成多个子任务，让不同 Agent 专注处理，再把结果汇总。

写一篇技术文章，可以拆成研究员、提纲师、写手、审稿人。

开发一个功能，可以拆成需求分析、代码实现、测试验证、代码审查。

关键不在"Agent 数量多"，而在每个 Agent 的职责边界足够清楚。

一句话：

**多 Agent 系统 = 多个单 Agent + 明确的协作协议。**

没有协议的多 Agent，只是一群模型互相发消息。

## 二、为什么需要多 Agent

- **上下文窗口有限**：一个 Agent 塞进所有资料、所有工具、所有角色说明，最后什么都懂一点，什么都做不稳。

- **专业化能提升质量**：研究员 Agent 的 Prompt 可以专门强调资料来源，审稿 Agent 可以专门找漏洞，执行 Agent 可以专门关注工具调用和错误恢复。

- **可以并行**：多个独立子任务同时跑，比一个 Agent 串行处理更快。

但多 Agent 不是越多越好。每多一个 Agent，就多一份上下文成本、一条协作链路、一个出错点。

简单任务用多 Agent，通常是过度设计。

## 三、三种经典拓扑

- **Orchestrator-Worker**。

一个总控 Agent 负责任务拆解、分派、汇总，多个 Worker 负责具体子任务。生产里最常见，也最稳，因为决策权集中，容易收敛。

- **Pipeline**。

任务按固定流程流转，比如检索 → 提纲 → 写作 → 审稿。适合流程稳定、阶段明确的业务。

- **Peer-to-Peer**。

多个 Agent 对等讨论，没有明确总控。demo 里最像"群体智能"，生产里最容易乱。

对等拓扑最大的问题是没人拍板。两个 Agent 意见冲突，谁说了算？讨论什么时候结束？成本谁控制？

工业级经验很简单：对等协作慎用。如果一定要辩论，外面也要套一层 Orchestrator 强制收敛。

## 四、协作的本质：状态还是消息

多 Agent 协作有两条路。

- **共享状态**：所有 Agent 读写同一个任务状态，例如 `research_notes`、`draft`、`review_comments`。好处是全局可见，适合状态机和长任务；坏处是并发写入和状态污染要管好。

- **消息传递**：Agent 之间通过明确消息互相交付结果。好处是边界清楚，容易追踪；坏处是上下文可能重复传递，协调成本高。

很多工业系统会混用：关键业务状态放共享 State，Agent 间交互用消息记录，所有变化进 trace。

无论哪种方式，最重要的是可追踪。你必须知道每个结论是谁生成的、基于什么上下文、传给了谁、有没有被验证。

多 Agent 没有 trace，调试会非常痛苦。

## 五、多 Agent 的几个生产杀手

- **幻觉传播**：一个 Agent 编错事实，后面的 Agent 把它当真，最后错误被包装得越来越像真的。

治法是跨 Agent 事实校验：关键事实必须带来源，汇总前做验证。

- **决策权争夺**：两个 Agent 都觉得自己该拍板，任务来回拉扯。

治法是明确最终裁决者，通常是 Orchestrator。

- **成本爆炸**：单 Agent 跑 5 轮，多 Agent 可能是 5 个 Agent 各跑 5 轮，成本瞬间乘起来。

治法是全局 token 预算，不是每个 Agent 各管各的。

- **调试地狱**：单 Agent 出错还能看一条 trace，多 Agent 出错如果没有边界 checkpoint，很难知道是哪个 Agent 开始跑偏。

治法是每个 Agent 入参、出参、工具调用、状态写入都结构化记录。

- **协调风暴**：Worker 遇到问题不断问 Orchestrator，Orchestrator 又不断补充，协调成本超过执行成本。

治法是一次性下发足够上下文，并限制回询次数。

## 六、收束

多 Agent 本质上不是 AI 问题，是分布式系统问题。

它关心的是一致性、协调、容错、追踪、成本控制。唯一新增的麻烦是：每个节点不是确定性的服务，而是一个可能瞎说的概率模型。

所以多 Agent 不是为了看起来高级，而是当单 Agent 已经被上下文、角色、工具、执行时长压垮时，用工程拆分换稳定性。
