# Prompt 不是咒语，是结构化的上下文注入

> 《工业级 AI Agent 开发实战》系列 · 第 03 篇

---

你在小红书、知乎、各种付费星球里能看到的典型内容：

"10 个能让 GPT 智商+50 的神奇开头"

"加上这句话输出质量立刻提升 200%"

"人格设定秘籍：这样写 Prompt 让 AI 把你当老板"

这些内容有一个共同特点：把 Prompt 说得像咒语——好像有某种神秘的触发词可以"激活"模型的隐藏能力。

我见过的 Prompt 付费课，95% 在卖的是这种感觉。而买过这些课的工程师，做出来的系统往往三天崩一次。

Prompt 没有那么神秘。它的本质就一句话：**Prompt 是在用一段结构化的文本，给一个无状态的概率模型，注入它做出期望输出所需的全部上下文。**

---

## 四种上下文，全部靠你自己塞进去

LLM API 是无状态的。每一次调用，模型看到的世界就是你发过去的那个 `messages` 数组，别的什么都没有。

所以模型做任何事，它需要的所有信息都必须在这段 messages 里。具体是四类：

**角色与行为约束**（Who）：你希望模型扮演什么角色、用什么口吻、不能做什么。通常放在 System Prompt 里。

**任务与目标**（What）：这次调用要干什么具体的事。通常在 User 消息里或 System 的末尾。

**背景知识与数据**（Context）：做这件事需要用到的信息——公司资料、检索到的文档、对话历史、工具返回结果。这一部分是工业级系统里占比最大的。

**输出规范**（How）：你希望模型吐出什么格式——纯文本？JSON？Markdown？按哪个 schema？

好的 Prompt 就是把这四类上下文拼成一段模型"读得懂、做得对"的文本。仅此而已。

所谓"Prompt 工程"，工程部分全部在这里：如何把这四类信息稳定、高效、低成本地装进 messages 数组，建立一套可维护、可评估、可复用的上下文组装流水线。这才是工业级 Prompt 系统要解决的问题。不是找对咒语，是搭建流水线。

---

## System Prompt 的"优先级"是怎么来的

很多文章会说"System Prompt 拥有最高执行优先级"，听起来像是模型里有一个 if-else 在强制执行 System 指令。

不是这样的。

真相是个统计现象：在模型训练时（特别是 RLHF / DPO 阶段），标注团队塞给模型大量"system 消息被严格遵守"的样本并用 RLHF 强化这种行为。训练完的模型因此形成了一种统计倾向——看到 system 消息里的内容，输出更倾向于符合它。

统计倾向，不是强制约束。这解释了两件事。

GPT-4 对 System 相当顽固，你硬塞"你现在不再是助手"通常拦不住它。但很多开源模型，尤其是微调得不够充分的，System 的约束力很弱——换个话术就绕过去了。所以你不能把任何安全性相关的规则全压在 System Prompt 上。线上系统里 Prompt 能挡掉 Prompt Injection 的比例大概在 60%-80%，剩下那部分必须靠独立的 Guardrail 层兜底。

另一件事：模型对 Prompt 的注意力并不是均匀分布的。有大量实验证明了一个现象叫"Lost in the Middle"——放在 Prompt 中段的内容，被模型有效利用的概率，比头部和尾部低得多。

所以最关键的硬性规则放头部，次要的风格说明放中段，当前这轮任务的具体指令放尾部。绝不要把核心规则放在一段长得没人会看完的 System 中间。

还有一个我见过很多次的浪费：把业务数据塞进 System Prompt。有个项目把公司 3000 字的产品手册全塞进 System——结果每次请求固定付费 1500 个 input token，一个月多烧了五位数。业务数据该走 RAG 检索后动态注入 user 消息，System 只装"如何做事的元规则"，不装"要用的材料"。

## Few-shot 的本质

"Few-shot learning"这个词严重误导人。它让人觉得你给模型看几个例子，它就"学会"了新能力。

模型在推理阶段是不会学习的，参数根本不会变。那几个例子真正做的事，是通过模式匹配，把模型的输出分布向你给出的格式拉齐。

说人话：你不是在教它，你是在给它一张模板。

假设任务是"从一段话里提取姓名和职位"。不用 few-shot，模型可能给你"在这段话中，人名是张伟，职位是产品经理"——多了废话，格式不稳。用了两个示例之后：

```
示例：
输入：王芳担任字节跳动的高级工程师。
输出：{"name": "王芳", "position": "高级工程师"}

输入：李娜是腾讯的设计总监。
输出：{"name": "李娜", "position": "设计总监"}

输入：张伟是上海银行的产品经理。
输出：
```

模型几乎必然返回 `{"name": "张伟", "position": "产品经理"}`。它不是"学会"了提取，而是看到前面的输入-输出对的模式，推断你后面要的也是这个格式——本质是一个高级的"填空游戏"。

工程上三点值得记：示例的多样性比数量更重要，3 个覆盖不同情况的示例比 10 个都差不多的要好；示例本身也在吃 token，能用 2-shot 解决的不要用 5-shot；示例的顺序会影响结果，把反例放前面、正例放后面，让最后的"好格式"成为模型最容易抄的那一个。

## 输出格式控制：结构化远胜文本解析

这是整个 Prompt 工程里工业级和玩具级差距最大的一条。

我见过很多项目这样做：

```
请回答用户的问题，并在最后一行用【意图：xxx】标注意图。
```

然后用正则 `r'【意图：(.+?)】'` 去解析。

这套东西在生产环境平均三天崩一次。模型心情不好的时候会写成"意图：xxx"（少了括号）、"意图是 xxx"（多了字）、"【intent: xxx】"（英文串味）、或者干脆忘了标。每一种变体都需要你加一条正则补丁，永远补不完。

工业级做法按优先级从高到低：

**首选 Tool Use / Function Calling**。你定义一个 JSON Schema，模型按 schema 吐结构化输出。不是"请求模型按格式回答"，是模型在协议层就必须吐 JSON。解析失败率从"每天若干次"降到"几乎为零"。

```python
tools = [{
    "type": "function",
    "function": {
        "name": "extract_intent",
        "parameters": {
            "type": "object",
            "properties": {
                "intent": {"type": "string", "enum": ["查询", "投诉", "退款", "其他"]}
            },
            "required": ["intent"]
        }
    }
}]
```

**其次是 JSON Mode / Structured Output**。OpenAI 的 `response_format={"type": "json_schema", ...}` 在解码阶段强制输出满足 schema 的 JSON，不需要你装成一个工具。

**最后才考虑文本 + 正则**。在旧模型、私有部署、不支持结构化输出的场景用，而且必须用 2-3 个 few-shot 示例锁住格式，解析时兜底逻辑拉满。

一条铁律：**任何业务代码要消费的 LLM 输出，必须是结构化的。不要让自然语言出现在你的业务逻辑里。** 你代码里任何一个 `split`、`regex`、`eval` 都是下一次生产事故的种子。

## CoT 不是万能药

Chain-of-Thought 在某些任务上确实能显著提升准确率——数学题、多步推理、逻辑判断。但它有两个被掩盖的代价。

代价一：Token 成本翻倍。CoT 每次都要让模型输出一大段"思考过程"，单次调用的 completion_tokens 轻松翻 2-3 倍。一个客服 Agent 加了 CoT，月账单直接乘 2 不是夸张。

代价二：在简单任务上反而有害。当任务本身不需要推理时，强行加 CoT 会让模型"想太多"。模型会给你编出一套推理过程来支持一个它其实本来就能直接答对的问题，推着推着自己把自己说服，跑偏了。我见过的数据是，在相当一部分简单分类和抽取任务上，加 CoT 会让准确率不升反降。

取舍的原则：复杂推理和多步任务用 CoT，但把思考过程塞到 tool_calls 的内部字段里不暴露给用户；简单分类、抽取、翻译不要用 CoT，直接结构化输出；成本敏感场景考虑小模型 + 不加 CoT，而不是大模型 + CoT。

## 真正管用的几条规律

讲了这么多，把 Prompt 工程里经得起工业级考验的规律提炼出来，只有这么几条，但每一条都能沉淀成代码：

**明确 > 礼貌**。"请用一句话（≤30字）总结下文的核心观点"，比"请帮我总结一下这段话"好得多。越明确的指令效果越稳。

**约束 > 允许**。告诉模型"不要做 X"经常不如告诉它"只能做 Y"。"不要输出代码以外的内容"不如"输出格式：只能是一个 Python 代码块，不含任何其他文字"。这是白熊效应——你叫人"不要想一只白熊"，他脑子里会立刻出现一只白熊。

**结构 > 散文**。用 Markdown 层级、XML 标签、JSON Schema 明确分块。Anthropic 官方建议对 Claude 使用 XML 标签（`<task>`、`<context>`、`<example>`），因为 Claude 在训练时就被喂过大量这种结构。不过要注意，这个建议对 GPT-4o 系列未必有同等效果，不要当成通用规律来套。

**先动态再静态**。System Prompt 是固定成本，每次都发。User 消息是按需。能放 user 就不要放 System，特别是那些只在某些场景才用到的规则或数据。

**Prompt 是代码，不是文案**。工业级项目里 Prompt 必须版本化（Git 管理，每次改动可追溯）、参数化（用模板引擎，不要拼字符串）、被测试覆盖（每次改 Prompt 跑回归集）、上线前评估（对比新旧版本在大量场景上的指标）。没有版本管理、没有回归测试的 Prompt，就是一段随时可能让生产环境崩掉的幽灵代码。

---

工业级 Agent 项目里，真正稀缺的不是"会写 Prompt 的人"，是建得起 Prompt 版本管理 / 评测 / 灰度 / 回滚这一套基础设施的工程师。一个合格的后端工程师加一套好的 Prompt 管理基础设施，就能干掉 90% 市面上打着"Prompt 专家"旗号的人。
