# 不用框架，200 行 Python 手写一个工业级雏形 Agent

> 《工业级 AI Agent 开发实战》系列 · 第 10 篇

前 9 篇讲完，你应该已经知道 Agent 不是一个神秘概念。

它底下是 LLM 调用、Prompt、Memory、RAG、Tool、循环、多 Agent 和 7 层架构。

但只看懂概念还不够。你必须亲手把它们粘起来一次。

这一篇的目标很简单：不用 LangChain，不用 LangGraph，不用 CrewAI，不用 AutoGen，用原生 Python 写一个能跑的 Agent 雏形。

注意，是不用 Agent 框架，不是不用 SDK。

OpenAI Python SDK 只是模型客户端，等价于更好用的 HTTP 封装。它不会替你做循环、工具注册、消息管理、记忆策略。我们要避开的是第三层 Agent 框架，而不是底层模型 SDK。

## 一、要写出的 Agent 长什么样

这个 Agent 至少要具备几件事。

- **能维护 messages**：因为 LLM API 是无状态的，历史上下文必须你自己塞回去。

- **能注册工具**：工具要有 schema、描述、参数和真实执行函数。

- **能跑 ReAct 循环**：每一轮模型要么返回最终答案，要么请求工具调用。

- **能处理终止条件**：最大轮数、token 预算、重复动作，都要显式控制。

- **能记录 trace**：线上出问题时，必须知道模型每一步为什么这么做、调了什么工具、工具返回了什么。

这已经不是玩具聊天脚本，而是一个工业级雏形。

## 二、第一步：工具层

先写工具，不要先写 Agent。

工具层至少要包含三样东西：

- 工具函数；
- 工具 schema；
- 工具注册表。

比如 `get_weather`，真实场景里会调天气 API；demo 里可以 mock。比如 `calculate`，可以用受限表达式求值，但不能直接裸 `eval`。

工具 schema 不是摆设。它告诉模型什么时候用这个工具、参数是什么、哪些字段必填。

这里最重要的工程习惯是：工具实现和工具描述放在一起管理。模型用错工具时，你要能快速修改描述和参数约束，而不是到处找 Prompt。

## 三、第二步：消息管理

LLM 不记得上一轮发生了什么。

你必须自己维护一组 messages：system 放规则，user 放用户输入，assistant 放模型历史输出，tool 放工具结果。

很多 Agent bug 都出在这里。工具结果没有按正确格式塞回去，模型下一轮就不知道刚才发生了什么；历史不做裁剪，token 很快爆；system prompt 里混入业务数据，后面很难维护。

消息管理就是 Agent 的短时记忆。

别小看它。所谓 Memory，本质上就是你每次请求时怎么伪造一段"模型好像记得"的历史。

## 四、第三步：ReAct 主循环

主循环的骨架很朴素：

- 把当前 messages 发给模型。

- 如果模型返回最终答案，结束。

- 如果模型返回 tool call，校验参数，执行工具。

- 把工具结果作为 tool message 追加进 messages。

- 继续下一轮。

这就是 Agent 的心脏。

但工业级雏形不能只写一个 `while True`。你要加最大迭代次数、token 预算、重复 tool call 检测、异常分类和 trace 记录。

没有这些防护，demo 会显得很聪明，上线会变成事故。

## 五、第四步：错误处理

LLM 调用是远程依赖，必须有超时、重试和错误分类。

工具调用也是远程依赖，也会失败。

错误不能只 `except Exception: pass`。你要告诉模型：这是临时失败、权限不足、参数错误，还是不可恢复错误。

否则模型会在错误里乱撞，反复调用同一个工具，或者基于错误结果编答案。

工业级 Agent 的复杂性，大量来自防御性编程，而不是模型本身。

## 六、第五步：Trace

没有 trace 的 Agent 不能上线。

用户说"刚才 AI 胡说八道"，你必须能回答：

- 当时 user 输入是什么；
- system prompt 是哪个版本；
- RAG 注入了哪些上下文；
- 模型决定调哪个工具；
- 参数是什么；
- 工具返回什么；
- 最后为什么生成这个答案。

Trace 不是为了好看，是为了定位责任边界。

模型错、工具错、记忆错、RAG 错、编排错，处理方式完全不同。

没有 trace，所有问题都会变成"AI 不稳定"。

## 七、LangChain 帮你省了什么

LangChain 这类框架确实省事。它帮你封装工具注册、消息拼接、AgentExecutor 循环、常用工具适配器。

但它也容易让新手误以为一行 `agent.run()` 就是生产系统。

默认配置必须逐项追问：

- `max_iterations` 是否合理；
- 工具异常怎么分类；
- 状态是否持久化；
- token 总预算谁控制；
- trace 是否能审计；
- 高危工具有没有 HITL。

这些框架不会自动替你想清楚。

框架真正的价值，是给你模板；生产级能力，需要你自己补齐。

## 八、收束

亲手写完这个 200 行左右的 Agent，你会有一个很强的感觉：Agent 工程的复杂性几乎全部来自传统软件工程。

模型决策只是一小部分。剩下的是消息管理、工具协议、错误处理、幂等、预算控制、终止条件、状态持久化、链路追踪。

这也是整个原理篇想讲清楚的事：

**Agent 工程不是 AI 创造了新工程，而是 AI 让旧工程的价值被重新激活。**

第一部分到这里结束。下一部分会进入进阶实践：RAG 进阶、Memory 进阶、LangGraph、CrewAI、Google ADK，以及同一个系统在不同框架里的实现对比。

但在那之前，建议你先把这个无框架 Agent 跑一遍。只有写过底层骨架，再看任何框架，你才知道它到底帮你省了什么，又默认漏掉了什么。
