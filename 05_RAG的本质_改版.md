# RAG 的本质：把搜索引擎塞进 Prompt 里

> 《工业级 AI Agent 开发实战》系列 · 第 05 篇

最近经常有人问：大模型上下文窗口越来越大，RAG 会不会死？

不会。

很多人把 RAG 理解成"窗口不够大时的临时补丁"。这个理解很危险。RAG 解决的从来不只是上下文长度问题，而是三个更根本的问题：知识太多、知识会变、知识必须按权限使用。

你把 1000 篇公司文档全塞进 Prompt，技术上可能能塞，工程上一定会烂。

模型不是数据库。它不会因为你把所有资料都发给它，就自动知道哪几段最重要。长上下文里注意力会衰减，成本和延迟会一起爆炸，知识更新还得每次重新注入。更麻烦的是权限：一个 HR Agent 不能因为"资料都在 Prompt 里"，顺手看到财务薪酬表。

所以 RAG 不会消失。只要 LLM 还是无状态模型，只要企业知识还是动态、分散、带权限的，RAG 就会一直存在。

## RAG 不是模型，是工程模式

RAG 的全称是 Retrieval-Augmented Generation，检索增强生成。

翻成人话就是：模型回答前，先让你的系统去翻资料，把最相关的几段塞进 Prompt，再让模型基于这些资料组织语言。

一个员工问："我入职满一年，年假有几天？"

朴素 LLM 只能凭训练记忆猜。RAG 系统会先去公司制度库里搜，找到"入职满 1 年至 5 年享受 5 天年假"这段，然后把它和用户问题一起发给模型。

所以 RAG 的本质不是"让模型变聪明"，而是**让模型别瞎猜**。

检索系统负责翻资料，LLM 负责说人话。RAG 的复杂性，几乎全部集中在"怎么翻得准"。

## 为什么不能直接塞所有文档

第一，注意力会衰减。长上下文不是无限精度内存。资料越多，模型越容易在无关段落里迷路，尤其是关键信息夹在中间时。

第二，成本和延迟会炸。每次问一个年假问题，都把整本员工手册、IT 手册、财务制度发给模型，这不是智能，是浪费。Token 成本按量计费，延迟也会跟着输入长度上涨。

第三，知识会变。企业制度、商品价格、库存、工单状态每天都在变。你不能指望模型训练时记住这些，也不能每次把全量知识重新拼一遍。

第四，权限必须前置过滤。RAG 不是"搜到了就塞"，而是"用户有权看的，才允许被搜到"。权限过滤如果放到模型回答之后，就已经晚了。

工业级 RAG 的第一原则：**不要让模型看到它不该看到的上下文**。

## 向量相似度的天花板

RAG 入门教程最爱讲 embedding 和向量库，好像向量检索就是 RAG 的全部。

不是。

向量相似度只能找"语义相似"，不能保证找"业务相关"。用户问"报销多久到账"，向量可能搜到"报销申请流程"，但真正相关的是"财务付款周期"。两段文字语义不一定近，业务上却强相关。

这就是朴素向量检索的天花板。

工业级 RAG 很少只靠 Top-K 向量搜索。常见做法是 Hybrid Search：关键词检索负责精确词、编号、人名、术语；向量检索负责语义相近；最后再用 reranker 做重排。

召回和答对不是一回事。召回只是把候选资料捞上来，重排才决定哪些资料值得进 Prompt。

## RAG 链路里真正难的是这些

切块决定上限。块太大，噪声多；块太小，上下文断。制度文档、代码文件、客服 FAQ、合同条款，切法都不一样。

Embedding 模型决定召回质量。同一套文档，只换 embedding 模型，召回准确率差 30% 很正常。中文业务、代码、法律、医学，各自适合的模型不同。

检索不是只取 Top-K。你要处理查询改写、同义词、关键词召回、时间过滤、权限过滤、多路召回合并。

Reranking 决定最终质量。第一轮召回宁可多捞一点，第二轮用 reranker 精排，把真正相关的段落排到前面。

注入也有工程问题。chunks 怎么排序？要不要带标题、来源、时间？多个文档冲突时谁优先？这些都会影响模型回答。

评估不能省。RAG 如果没有评测集、没有 faithfulness、没有 bad case 回归，本质上就是凭感觉调参。

## 朴素 RAG 的进阶路线

第一阶段：能跑。文档切块、embedding、向量库、Top-K、塞 Prompt。

第二阶段：能用。加 Hybrid Search、reranker、权限过滤、引用来源、无答案拒答。

第三阶段：可运营。加 RAGAS 或自建评估集，监控召回率、忠实度、拒答率，把用户 bad case 回灌进测试集。

第四阶段：领域优化。做查询改写、HyDE、领域 embedding 微调、GraphRAG、知识版本管理。

大多数团队卡在第二阶段，以为自己做了 RAG，其实只是把搜索结果塞进 Prompt。

真正的工业级 RAG，是一套完整的信息检索系统，只是最后一步接了 LLM。

## 收束

把今天的内容压成一句话：

**RAG 不是 AI 魔法，是搜索工程在 LLM 时代的复用。**

它解决的不是"模型记不住"，而是"模型不该自己记"。企业知识应该存储在可更新、可检索、可权限控制、可评估的系统里，而不是寄希望于模型训练参数。

所以 RAG 不会因为上下文窗口变大而消失。窗口越大，越需要更好的检索和注入策略，否则你只是把更多垃圾塞给模型。
