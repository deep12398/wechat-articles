# Function Calling 的本质：让模型先想清楚再动手

> 《工业级 AI Agent 开发实战》系列 · 第 06 篇

一个只能聊天的 LLM，不叫 Agent，最多叫聊天机器人。

Agent 真正和聊天机器人拉开差距的地方，是它能动手：查数据库、调 API、改文件、发邮件、创建工单、执行代码。

但这里有一个误解：不是模型真的执行了代码。模型从来没有"手"。

Function Calling 的本质是：模型决定"该调用哪个函数、参数是什么"，然后**你的代码**真的去执行这个函数，再把结果喂回模型。

换句话说，模型像一个不能碰键盘的经理。它能判断该让谁做事、怎么做，但真正按按钮的人，是你的后端程序。

## Tool 到底是什么

Tool 就是把一段真实世界能力包装成模型能理解的接口。

用户问："今天上海天气怎么样？"

模型不会自己访问天气 API。它只会输出一个结构化决策：调用 `get_weather`，参数是 `{"city": "Shanghai"}`。

你的后端接到这个决策，真的去调天气 API，拿到结果，再把结果作为 tool message 发回模型。模型最后把"22 度，多云"组织成自然语言给用户。

所以一句话：

**Tool = 让 LLM 在合适的时机，用合适的参数，让你的代码替它执行某个动作。**

工具调用里真正危险、真正复杂、真正需要工程能力的部分，都在你的代码里。

## Function Calling 是两轮交互

第一轮：你把用户问题、可用工具 schema 一起发给模型。模型不直接回答，而是返回 `tool_calls`。

中间：你的代码解析 `tool_calls`，校验参数，检查权限，执行真实函数，处理超时和错误。

第二轮：你把工具执行结果再发给模型。模型基于结果生成最终回答。

这就是 Function Calling 最核心的机制：**模型负责决策，代码负责执行，模型再负责解释结果。**

新手最容易踩几个坑。

第一个坑，`tool_calls` 可能是多个。模型可能一次决定并行查天气、查日历、查库存，你的执行层要支持并发和结果聚合。

第二个坑，参数可能不合法。模型输出 JSON，不等于 JSON 一定符合业务规则。类型、枚举、范围、权限都要你的代码校验。

第三个坑，工具会失败。API 超时、数据库报错、权限不足，都必须翻译成模型能理解的结构化错误，而不是直接把 stack trace 塞回去。

第四个坑，副作用工具必须幂等。创建订单、退款、发邮件这类动作，不能因为模型重试就执行两次。

## 描述比实现重要

后端工程师的本能是把函数实现写扎实。

但在 Agent 系统里，决定模型会不会用对工具的，首先不是函数实现，而是工具描述。

你写一个 `search`，描述是"搜索信息"，模型一定乱用。它不知道什么时候该搜，什么时候不该搜，搜不到怎么办，参数怎么填。

好的工具描述必须说清四件事：什么时候用、什么时候不用、参数怎么填、返回结果怎么理解。

描述越像 API 文档，模型越容易用错；描述越像操作手册，模型越容易用对。

这是反直觉的地方：工具实现写完可能就稳定了，但工具描述会随着 bad case 不断迭代。每发现一次模型用错工具，都是 schema 还没讲清楚。

工业级 Agent 团队大量调试时间，其实都花在工具描述上。

## Skill 比 Tool 高一层

单个 Tool 是一个动作，比如查天气、发邮件、读文件。

但真实工作很少是一个动作。写周报、处理退款、生成投放报告，往往需要一组工具、模板、规则和操作步骤。

Skill 解决的就是这个问题：把一组 Tool、一段引导 Prompt、一些资源文件打包成一个能力单元。

Tool 是动词，Skill 是一件完整工作。

如果一个任务只需要单步动作，用 Tool。只要任务涉及稳定流程、多个工具组合、固定资源模板，就该抽成 Skill。

这也是为什么 Skill 会变成 Agent 工程里的重要抽象。它不是为了炫技，而是为了降低模型每次临场组合工具的概率。

## MCP 的本质

在 MCP 出现之前，每个 Agent 框架接每套工具都要重写一遍。

Slack 接 LangChain 写一套，接 Claude Desktop 写一套，接自研 Agent 又写一套。N 个 Agent 乘以 M 个工具系统，就是 N × M 个集成。

MCP 要解决的是这个集成爆炸。

工具方实现 MCP Server，Agent 方实现 MCP Client。只要双方遵守协议，工具就能被不同 Agent 发现、读取 schema、调用。

这件事的意义不是"又多了一个协议"，而是让工具像 USB 一样变成可插拔基础设施。

对 Agent 工程师来说，未来不会写工具集成还不够，至少要知道怎么把企业内部系统包装成 MCP Server。

## 工具设计的几条铁律

工具粒度要不大不小。太细，模型要连续决策很多步，成功率下降；太粗，工具变成黑盒，模型无法灵活组合。

参数要极简。能从上下文推出来的，别让模型填。每多一个参数，就是多一个填错点。

返回必须结构化。不要返回一段自然语言让模型猜，返回 JSON，字段含义稳定。

副作用工具必须有 `idempotency_key`。所有会写库、发消息、扣钱、删数据的工具，都要能防重复执行。

错误要可读。给模型看的错误不是日志，是下一步决策依据。`permission_denied`、`timeout_retryable`、`invalid_argument` 要区分清楚。

高危操作必须 HITL。模型可以建议删除，不能自己删除。人类审批链路是生产系统的底线。

## 收束

工具层是 Agent 工程里最纯粹的传统软件工程主场。

这里不靠玄学，靠 API 设计、错误处理、幂等性、权限、审计、协议标准化。

模型可以换，框架可以换，但一套设计良好、描述打磨过、带幂等和审计的工具层，是很难替代的资产。
