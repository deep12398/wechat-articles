# Agent 循环的本质：一边想，一边动，一边修正

> 《工业级 AI Agent 开发实战》系列 · 第 07 篇

前面几篇我们把 Agent 的零件拆开讲了：LLM、Prompt、Memory、RAG、Tool。

但零件摆在桌上，不等于机器能跑。

Agent 真正开始像 Agent，是从循环开始的。用户给一个目标，模型判断下一步要做什么，你的代码执行动作，结果再回到模型，模型根据结果继续判断。

听起来像"AI 在思考"。

工程上，它就是一个 while 循环。

## 一、Agent 循环到底是什么

一个普通 LLM 调用是单轮的：用户问，模型答。

Agent 循环是多轮的：用户给目标，模型可能不直接答，而是先决定调用工具。工具返回结果后，模型再决定下一步。直到任务完成、预算耗尽、出错，或者触发终止条件。

最朴素的结构就是：

用户目标 → LLM 决策 → Tool 执行 → Observation 回填 → LLM 再决策。

所以一句话：

**Agent 循环 = 一个反复调用 LLM 的 while 循环。每一轮模型要么给最终答案，要么请求你的代码再执行一个动作。**

这就是 AgentExecutor、ReAct、LangGraph 状态机背后共同的骨架。

## 二、ReAct 的本质

ReAct 是 Reason + Act。

经典结构是 Thought、Action、Observation：先想一下，再行动，再看结果。

很多人把 Thought 当成废话，觉得模型自己在"自言自语"。但 Thought 的价值不是给人看，而是改变模型下一步输出分布。

让模型在调用工具前先明确"我为什么要调这个工具"，会降低乱调工具的概率。

Function Calling 时代，Thought 不一定要暴露给用户，也不一定要作为明文输出，但这个机制仍然存在。你可以让模型内部先规划，再输出结构化 tool call。

本质没变：先把任务状态整理清楚，再行动。

## 三、Thought 为什么有用

LLM 不是符号推理机，它是在给下一个 token 采样。

当你要求它先写出"我需要先查天气，因为用户问的是实时信息"，后面的工具选择会被这段上下文强烈影响。

这不是魔法，是上下文注入。

但 Thought 也有代价。它会增加 token 成本，也可能泄露不该给用户看的中间推理。生产系统里通常不会把完整 Thought 直接展示给用户，而是保留在 trace 里，或者只让模型输出短理由。

工业级做法不是迷信 CoT，而是让模型在关键决策点有足够的中间状态。

## 四、AgentExecutor 替你做了什么

框架里的 AgentExecutor 通常帮你做几件事：维护 messages，解析模型输出，执行工具，把 observation 塞回去，控制最大轮数。

这确实省事。

问题是，默认配置大多是 demo 级的。

`max_iterations` 设太大，循环失控；工具异常没分类，模型不知道下一步该重试还是放弃；状态不持久化，请求断了任务就丢；token 预算没有总控，几轮下来成本失控；trace 不完整，线上出问题无法复盘。

很多团队第一次上线 Agent 后，救火时间都花在补 AgentExecutor 的洞。

框架给的是循环模板，不是生产级循环。

## 五、终止条件是硬骨头

Agent 循环最容易被低估的部分，是怎么停下来。

正常完成当然要停。但更多时候，系统要在不确定状态下安全停止：工具连续失败、模型反复调同一个工具、token 快用完、用户目标已经偏离、外部依赖超时。

工业级至少要有四层终止防护。

- **第一层：最大迭代次数**。不要让 Agent 无限跑。

- **第二层：总 token 预算**。不是每轮预算，而是整个任务预算。

- **第三层：重复动作检测**。同一个工具、同一组参数反复调用，基本就是循环卡住。

- **第四层：业务状态机**。某些业务流程只能按合法状态跳转，不能让模型自由发挥。

终止之后也要设计行为：是给用户部分结果、请求人工接管、进入补偿流程，还是写入待恢复队列。

Agent 不只是会跑，关键是会安全地停。

## 六、ReAct 和 Plan-and-Execute 怎么选

- **ReAct**：适合路径不确定的任务。比如客服问答、探索性检索、用户意图不明确的场景，每一步都要根据新观察调整。

- **Plan-and-Execute**：适合路径清楚的任务。比如生成报告、数据分析、代码修改，先规划步骤，再逐步执行和校验。

代码助手通常是混合模式：先做计划，决定读哪些文件、改哪些模块、跑哪些测试；每一步内部再用 ReAct 处理细节。

长任务不要只靠 ReAct。步数越多，越容易漂移，必须引入显式计划、状态机和 checkpoint。

这也是 LangGraph 这类状态机框架在复杂任务里更稳的原因：它把"下一步该去哪"从纯模型决策，变成了模型决策 + 程序状态约束。

## 七、收束

Agent 之所以叫 Agent，不是因为它会思考，而是因为它能在循环中根据中间结果动态调整下一步动作。

这个能力听起来很 AI，落到工程里却是 99% 的传统后端问题：终止条件、错误恢复、状态持久化、预算控制、链路追踪。

模型决策当然重要，但循环工程更重要。循环本身设计不稳，再聪明的模型也会把系统带进死胡同。
